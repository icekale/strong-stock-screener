# 竞价复盘与规则优化设计

## 背景

竞价雷达当前用于早盘快速筛选：系统在 09:14:30 至 09:30:30 自动采样，并展示 09:20、09:23、09:24:50、09:25 四个关键时间点。但这些快照目前主要保存在内存缓存里，重启后不可回溯，也没有盘后和次日表现归因。

本功能把竞价雷达升级为可学习的闭环系统：每天稳定保存竞价样本，盘后和次日计算表现，用历史样本总结哪些竞价规则有效、哪些规则容易失败。

## 目标

- 从现在开始，每个交易日自动归档竞价关键时间点快照。
- 尝试补历史竞价样本；如果数据源无法提供历史集合竞价，则明确标注不可补，不伪造历史。
- 对每只竞价候选做三段归因：10:00 强度、当日结果、次日反馈。
- 以综合评分作为规则优化成功标准，同时在 UI 中拆开展示三段结果。
- 按竞价规则分桶统计命中率、收益、回撤、次日溢价和失败样本。
- 给出可执行的规则优化建议，例如降低高开过热权重、提高行业集中和量能确认权重。

## 非目标

- 第一版不做自动调参上线，不让系统自己改交易规则。
- 第一版不做实时交易提醒推送；通知和任务流放到后续版本。
- 第一版不承诺补齐历史竞价数据；历史回填取决于 TickFlow 或补充源是否能提供可验证数据。
- 第一版不保存完整全市场原始行情序列，只保存竞价候选、关键快照和归因指标，避免数据无限膨胀。

## 成功标准

综合评分由三段组成：

- 10:00 强度：09:25 后至 10:00 的最高涨幅、是否站稳、是否快速回落。
- 当日结果：当日最高涨幅、收盘涨幅、最大回撤、是否涨停。
- 次日反馈：次日开盘溢价、最高溢价、收盘溢价、是否延续强势。

综合评分用于规则统计排序，UI 必须同时展示三段拆分结果，避免单一分数掩盖风险。

## 数据模型

新增竞价复盘模型：

- `AuctionReviewRecord`
  - `trade_date`
  - `symbol`
  - `name`
  - `industry`
  - `selected_at_label`: `09:20`、`09:23`、`09:24:50`、`09:25` 或 `manual`
  - `selected_at`
  - `auction_snapshot`: 竞价涨幅、当前涨幅、成交额、换手、竞价评分、排名、tier、signals、risk_flags、quote_time
  - `rule_tags`: 从 signals、risk_flags、tier 和行业集中度生成的规则标签
  - `source_status`: 数据源状态和 fallback 信息
  - `intraday_result`: 10:00 前最高涨幅、10:00 涨幅、10:00 回落幅度、是否站稳
  - `day_result`: 当日最高涨幅、收盘涨幅、最大回撤、是否涨停
  - `next_day_result`: 次日开盘涨幅、最高涨幅、收盘涨幅、是否继续强势
  - `score`: 综合评分和三段拆分分数
  - `review_status`: `pending`、`intraday_done`、`day_done`、`next_day_done`、`data_incomplete`

- `AuctionRuleBucket`
  - `rule_tag`
  - `sample_count`
  - `win_rate`
  - `avg_score`
  - `avg_intraday_peak_pct`
  - `avg_close_pct`
  - `avg_next_open_pct`
  - `avg_drawdown_pct`
  - `failure_count`
  - `suggestion`

## 存储设计

新增 `AuctionReviewStore`：

- 存储目录：`STRONG_STOCK_DATA_DIR/auction_reviews`
- 每日样本：`records/YYYY-MM-DD.jsonl`
- 最新总结：`latest_summary.json`
- 规则统计：`rule_buckets.json`
- 保留策略：默认保留 120 个交易日，可通过配置项调整。

写入规则：

- 竞价采样命中关键时间点时，保存该时间点 Top 候选。
- 手动刷新也允许写入，但以 `trade_date + symbol + selected_at_label` 去重。
- 每个时间点默认保留前 100 只候选，和当前竞价页面一致。
- 不保存完整 K 线和全市场原始报价，只保存用于复盘和归因的字段。

## 归因流程

第一版提供手动触发 API，后续再接后台任务流：

- `POST /api/auction/review/finalize?trade_date=YYYY-MM-DD`
  - 读取当天竞价样本。
  - 拉取每只股票当日和次日 K 线。
  - 如果分钟线可用，计算 10:00 前最高涨幅和 10:00 表现。
  - 如果分钟线不可用，10:00 强度标记为 `data_incomplete`，仍计算日线和次日结果。
  - 保存每只股票归因和当天总结。

- `GET /api/auction/review/latest`
  - 返回最新竞价复盘总结。

- `GET /api/auction/review?trade_date=YYYY-MM-DD`
  - 返回指定交易日复盘记录和分桶统计。

- `GET /api/auction/rules/summary`
  - 返回跨日期规则分桶统计。

## 历史回填

新增 `POST /api/auction/review/backfill`：

- 输入日期范围和最大天数。
- 优先尝试 TickFlow 是否能提供历史集合竞价或历史分钟级早盘数据。
- 如果不能获得可验证的历史集合竞价数据，返回 `data_unavailable`，并说明原因。
- 不用当日 K 线反推竞价信号，不做“看未来”的伪回测。

## 规则分桶

第一版内置规则标签：

- `强势高开`: open_gap_pct >= 7
- `温和高开`: 3 <= open_gap_pct < 7
- `低开转强`: open_gap_pct <= -3 且 current_pct_change >= 3
- `低开偏弱`: open_gap_pct <= -3 且 current_pct_change <= -3
- `量能活跃`: turnover_cny >= 300,000,000 或 turnover_rate >= 5
- `成交额居前`: turnover_cny >= 1,000,000,000
- `高开过热`: risk_flags 包含高开回落风险
- `行业集中`: 当前时间点 Top 行业集中度较高

规则建议按样本统计生成，不直接改生产规则：

- 高分高胜率：提高观察优先级。
- 高分低胜率：降低权重或增加过滤条件。
- 高回撤：标记风险提示。
- 样本不足：展示“样本不足，不建议调整”。

## UI 设计

在 `/auction` 页面新增 `竞价复盘` 区块，保持早盘主操作不变：

- 今日复盘
  - 显示当天关键时间点是否已归档。
  - 显示待归因、已归因、数据不完整数量。
  - 提供“生成/刷新今日复盘”按钮。

- 历史日期
  - 按日期查看当天竞价候选。
  - 展示每只股票三段结果和综合评分。

- 规则统计
  - 展示规则分桶表：样本数、胜率、平均收益、平均回撤、次日溢价、建议。
  - 支持按行业、规则标签过滤。

- 失败样本
  - 显示高分失败股票。
  - 用于复盘为什么竞价信号失效。

## 错误处理

- 数据源缺失：返回明确 source status，不生成伪结果。
- 分钟线权限不足：仍生成日线/次日复盘，10:00 强度标记为缺失。
- 次日数据未出现：状态保留为 `day_done`，等待下一交易日补全。
- 重复归档：按唯一键去重，避免多次刷新导致文件膨胀。
- 存储膨胀：按保留天数清理旧 JSONL。

## 测试策略

后端测试：

- 竞价快照保存为 JSONL，按 `trade_date + symbol + selected_at_label` 去重。
- 关键时间点归档后，重启 store 仍能读取历史。
- 归因计算正确处理 10:00 强度、当日结果、次日反馈。
- 分钟线缺失时不失败，状态为 `data_incomplete`。
- 规则分桶能统计样本数、胜率、平均收益和建议。
- 保留策略能删除旧交易日文件。

前端测试：

- API 类型包含竞价复盘模型。
- `/auction` 页面包含竞价复盘入口、规则统计、失败样本和刷新按钮。
- 页面不影响现有竞价雷达、时间轴、行业快速筛选。

## 后续版本

- 交易时段任务流自动归因和通知。
