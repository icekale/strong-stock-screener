# 自用专业版工作台硬化设计

## 背景

StockMaster 已经从单一强势股筛选器扩展成完整短线工作台，包含选股、竞价、板块、情绪、自选股、个股 K 线、模型维护和数据源设置。功能已经覆盖实战流程，但当前体验仍有明显试验阶段特征：

- 首页、竞价、板块、情绪、自选股和设置页的操作层级不完全一致。
- 部分页面仍可能在用户打开或刷新时触发重型实时数据请求。
- 后端有 TTL cache 和采样线程，但缺少统一的数据快照层、任务状态层和慢接口观测。
- 出错时用户经常只看到 `500 Internal Server Error`，不知道是 TickFlow、iFinD、TDX、fallback 还是本地缓存的问题。
- 主流程和维护/调试功能混在一起，日常实战时需要自己判断下一步该看什么。

本设计按用户确认的方向执行：先做“自用专业版”，不优先做 SaaS 化、多用户或商业包装。目标是让系统成为每天早盘和盘中可以稳定依赖的交易辅助工作台。

## 目标

1. 提升实战稳定性：页面优先读取本地快照，避免每次刷新直接打全市场实时接口。
2. 提升速度：核心页面缓存加载达到可接受预算，手动刷新异步执行并显示进度。
3. 提升可信度：所有关键数据展示缓存年龄、来源、fallback、最近错误和生成时间。
4. 收敛产品主流程：把系统组织成盘前准备、竞价决策、盘中确认、盘后复盘四个场景。
5. 降低试验感：统一页面模板、加载态、空态、错误态、按钮层级和状态文案。
6. 保持单用户私有部署：继续适配 Docker Hub 和 Unraid，不引入账号系统。

## 非目标

- 不做登录、多用户、权限、订阅计费或公网 SaaS。
- 不重写选股策略、竞价模型或 GSGF 规则。
- 不把旧的每日大盘复盘项目重新合并进来。
- 不让 AI 自动修改策略参数或直接生成交易指令。
- 不为了视觉统一一次性重写所有页面组件。
- 不引入重量级外部任务队列，第一阶段仍以单容器可部署为约束。

## 推荐方案

采用“产品硬化 + 性能治理”的渐进方案：

1. 第一阶段先做统一快照和缓存状态，解决页面慢、偶发 500、数据来源不透明。
2. 第二阶段重塑首页和导航，把主流程收敛成日常工作台。
3. 第三阶段统一 UI 状态和设置页系统控制台，让页面像一个完整产品。
4. 第四阶段做工程治理和性能观测，拆分过大的后端入口，补齐 smoke 测试。

不推荐现在做平台化。当前最高价值是让单用户实战链路稳定、快速、可解释。

## 产品结构

### 四个实战场景

系统主导航按实战场景解释，而不是按技术模块解释：

- 盘前准备：自选观察池、昨日复盘、待关注题材、模型健康。
- 竞价决策：竞价强度榜、Top3 试运行、行业/题材集中、风险与观察。
- 盘中确认：短线情绪、板块曲线、涨跌停结构、成交额排行。
- 盘后复盘：筛选记录、竞价复盘、模型维护、自选股复盘结论。

现有页面可以继续保留，但首页只承担“今日工作台”角色，不再把所有模块堆到同一个长页面。

### 今日工作台

首页调整为高频入口，默认展示：

- 今日状态：交易日、当前时段、数据源健康、缓存年龄。
- 今日动作队列：现在最该做的 1 到 3 个动作，例如运行筛选、查看竞价、更新自选观察。
- 最新快照：竞价、情绪、板块、选股结果的摘要状态。
- 自选风险提醒：观察池中今日触发风险或动量变化的股票。
- 最近一次筛选结果：保留核心结果和进入详情的路径。
- 模型维护入口：只显示健康状态和待确认建议数量。

维护、调试、校准、手动补数据等功能统一收进“高级维护”或对应详情页。

## 后端架构

### 统一快照层

新增 `MarketSnapshotService`，负责聚合并保存页面所需的高频市场状态。第一版覆盖：

- 首页市场状态。
- 竞价快照和 09:25 锁定结果。
- 短线情绪摘要。
- 板块/题材工作台摘要。
- 数据源健康摘要。

快照统一包含：

- `snapshot_id`
- `trade_date`
- `generated_at`
- `expires_at`
- `cache_age_seconds`
- `source_status`
- `data_quality`
- `payload`

页面默认读快照。只有用户点击“刷新数据”或后台采样触发时，才调用 provider 重新生成。

### 数据源状态结构

关键接口统一返回数据源元信息：

- `primary_source`
- `fallback_source`
- `used_fallback`
- `latency_ms`
- `cache_hit`
- `missing_count`
- `last_success_at`
- `last_error_at`
- `last_error`
- `confidence`: `fresh | stale | partial | degraded | unavailable`

前端不再只显示“成功/失败”，而是显示“可用但过期”“部分 fallback”“不可用”等状态。

### 后台任务和采样

保留单容器部署，不引入 Redis/Celery。先在现有线程模型上加一层统一任务注册：

- `auction_sampler`: 竞价时段高频采样。
- `sector_sampler`: 交易时段板块/题材采样。
- `sentiment_sampler`: 交易时段情绪采样。
- `snapshot_refresher`: 首页快照定时刷新。
- `maintenance_jobs`: 模型维护、校准和复盘。

每个任务暴露：

- 当前状态。
- 上次成功时间。
- 上次失败时间。
- 最近错误。
- 当前间隔。
- 最近一次耗时。
- 下一次计划运行时间。

新增系统状态接口：

- `GET /api/system/status`
- `GET /api/system/jobs`
- `GET /api/system/cache`
- `POST /api/system/cache/clear`

### 缓存策略

现有 `TtlCache` 需要硬化：

- 避免在全局锁内执行慢 `factory`。
- 对同一个 key 做请求合并，避免并发刷新打爆 provider。
- 支持返回 stale cache，同时后台刷新。
- 记录 hit/miss、刷新耗时、刷新错误。
- 支持按 namespace 清理缓存。

建议缓存层分三类：

- 内存短 TTL：实时行情、板块摘要、情绪摘要。
- 本地快照文件：竞价锁定、板块采样、情绪采样、筛选记录。
- 模型/计算结果缓存：竞价 Top3、GSGF 校准、AI 模型维护报告。

### 性能预算

第一版按以下目标验收：

- 首页缓存加载小于 800ms。
- 竞价页缓存加载小于 500ms。
- 板块页缓存加载小于 1000ms。
- 情绪页缓存加载小于 1000ms。
- 个股 K 线缓存命中小于 500ms。
- 手动刷新可以超过 3s，但必须显示任务状态、耗时和数据源。
- 核心页面失败时不白屏，展示上一份可用快照和错误原因。

## 前端设计

### 页面模板

所有主页面统一结构：

1. 标题区：页面用途、交易日、数据新鲜度。
2. 状态区：数据源、缓存年龄、最近错误、刷新按钮。
3. 主操作区：本页面最重要的 1 到 2 个操作。
4. 结果区：表格、图表或股票列表。
5. 辅助区：解释、复盘、维护入口。

### 状态组件

新增统一组件：

- `DataFreshnessBadge`
- `SourceStatusStrip`
- `AsyncJobProgress`
- `SnapshotFallbackNotice`
- `WorkbenchEmptyState`
- `WorkbenchErrorState`

所有页面复用这些组件，避免每个页面各写一套提示。

### 设置页控制台

设置页升级为“系统控制台”，分区如下：

- 数据源：TickFlow、iFinD、TDX、东方财富 fallback。
- AI：OpenAI-compatible、DeepSeek、模型名称、测试调用。
- 通知：Telegram、企业微信、飞书、邮件。
- 后台任务：采样器、模型维护、最近运行状态。
- 缓存与存储：缓存年龄、快照数量、保留策略、清理按钮。
- 版本信息：Git commit、Docker tag、构建时间、时区。

这能减少 Unraid 上排查问题时的盲区。

## 错误处理

后端错误分级：

- `provider_error`: 上游数据源失败。
- `provider_timeout`: 上游超时。
- `rate_limited`: 被限流或 429。
- `cache_miss`: 本地无可用快照。
- `partial_data`: 数据不完整但可展示。
- `internal_error`: 本地逻辑错误。

前端展示规则：

- 有旧快照：显示旧快照，同时提示数据已过期和刷新失败原因。
- 无旧快照：显示明确空态和下一步操作。
- provider 限流：提示等待后台采样，不鼓励连续手动刷新。
- 内部错误：显示错误编号和最近日志摘要，便于排查。

## 工程治理

### 后端拆分

当前 `apps/api/app/main.py` 已经承担过多职责。按路由和服务边界逐步拆分：

- `routers/screener.py`
- `routers/auction.py`
- `routers/sectors.py`
- `routers/sentiment.py`
- `routers/stocks.py`
- `routers/settings.py`
- `routers/system.py`
- `services/market_snapshot.py`
- `services/cache_registry.py`
- `services/provider_health.py`
- `services/job_registry.py`

拆分时保持 API 路径兼容，避免前端和部署一次性大改。

### 日志和观测

新增结构化请求日志：

- route
- status_code
- duration_ms
- cache_hit
- provider
- provider_latency_ms
- fallback
- error_type

新增慢接口阈值：

- API 超过 1000ms 记录 warning。
- Provider 超过 3000ms 记录 warning。
- 交易时段连续失败 3 次记录 degraded。

### 测试

后端：

- 快照服务生成、读取、过期、fallback 测试。
- cache 请求合并和 stale return 测试。
- 系统状态 API 测试。
- provider 错误分类测试。
- 主路由 smoke 测试。

前端：

- 首页状态组件测试。
- 设置页系统控制台测试。
- stale/error/empty 状态渲染测试。
- 竞价、板块、情绪页面缓存优先读取测试。

端到端：

- 本地 `http://127.0.0.1:3110/` 主页面打开。
- `/auction`、`/sectors`、`/sentiment`、`/watchlist`、`/settings` 不白屏。
- Unraid 实例健康检查和主要 API 不返回 500。

## 分阶段计划

### 阶段一：性能底座

- 新增 cache registry 和系统状态 API。
- 硬化 `TtlCache`。
- 建立首页、竞价、情绪、板块的快照元信息。
- 前端展示缓存年龄、来源、fallback 和最近错误。

验收标准：

- 核心页面缓存命中时达到性能预算。
- provider 失败时页面仍展示上一份快照。
- 设置页可以看到后台任务和缓存状态。

### 阶段二：今日工作台

- 重构首页为“今日工作台”。
- 收敛主操作和动作队列。
- 将维护/调试入口移到高级区域。
- 最近筛选结果、自选风险、竞价状态和板块状态作为摘要卡出现。

验收标准：

- 首页打开后用户能立刻知道今天该做什么。
- 首页不再主动触发多个重型实时请求。
- 选股结果、竞价、板块、情绪入口清晰。

### 阶段三：系统控制台和统一状态

- 设置页改为系统控制台。
- 统一 loading、empty、error、stale 和 degraded 状态组件。
- 所有主页面复用 `SourceStatusStrip` 和 `DataFreshnessBadge`。

验收标准：

- 出错时能判断是哪个数据源、哪个任务、哪个缓存出了问题。
- 页面状态视觉和文案一致。

### 阶段四：后端拆分和 smoke 测试

- 按 router 拆分 `main.py`。
- 增加请求耗时日志和慢接口统计。
- 增加本地和 Unraid smoke 测试脚本。
- 更新 README 和发布检查清单。

验收标准：

- 后端文件边界更清楚。
- 新功能不再继续堆到 `main.py`。
- 每次推送 Docker Hub 和 Unraid 前有固定检查。

## 风险和取舍

- 快照优先会带来“数据不是绝对实时”的问题，因此必须清楚展示缓存年龄和刷新状态。
- 单容器内线程任务足够当前自用专业版，但不是大规模多用户架构。
- 首页重构可能改变使用习惯，需要保留原模块入口。
- 不建议一次性拆完 `main.py`，否则风险超过收益。

## 成功标准

这次产品化完成后，系统应该达到：

- 早盘打开竞价页，不需要等待长时间实时计算。
- 首页像交易工作台，不像功能试验合集。
- 数据慢或失败时，用户知道原因和可信度。
- Unraid 上出现问题时，可以在设置页直接定位数据源、缓存、任务或版本。
- 后端性能问题可以通过日志和系统状态复盘，而不是靠猜。
