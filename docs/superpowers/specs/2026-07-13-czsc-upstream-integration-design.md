# CZSC 上游集成与选股影子验证设计

## 背景

项目当前已经具备独立的多周期缠论工作台、日线/60 分钟/30 分钟/5 分钟结构分析、回放、回测、预警、模拟盘，以及候选股的缠论摘要与共振分。当前稳定依赖为 `czsc==0.10.12`，适配器直接消费 `CZSC` 的分型和笔，其余线段、中枢、背驰、买卖点和多周期共振主要由项目规则层派生。

上游 `waditu/czsc` 在 `1.0.0rc8` 完成了 Rust/PyO3 重写。核心对象、信号注册方式和调用接口均有破坏性变化，250 多个信号函数已迁移到原生模块。直接替换当前依赖会同时影响图表、筛选、回放、预警和模拟盘，不适合作为一次普通依赖升级。

本设计采用双运行时隔离和影子验证：保持 `0.10.12` 为正式路径，在独立 Python 环境运行 `1.0.0rc8`，将上游精选信号映射为项目自有证据模型。工作台可立即展示研究信号，选股器只计算 `czsc_score_v2` 和影子排名。任何正式切换都必须经过五年滚动验证和人工确认。

上游研究基于以下版本：

- 仓库：`waditu/czsc`
- 上游提交：`9ab62854f6bfab8515115b942baf4deb0f06185c`
- 包版本：`1.0.0rc8`
- 许可证：Apache-2.0

## 目标

1. 在不改变现有正式缠论结果的前提下接入 `1.0.0rc8` 的增量结构与精选信号能力。
2. 在 `/chanlun` 工作台展示可追溯、可开关且不会与正式信号混淆的上游研究证据。
3. 为强势股候选计算解释明确的 `czsc_score_v2`、可用性和影子排名，但不改变正式排序、筛选、预警或模拟订单。
4. 使用严格的已收盘 K 线和逐根回放消除未来函数，完成五年滚动时间切分验证。
5. 提供可复现的数据集构建、验证 CLI、机器可读报告和审计清单。
6. 保持正式工作台和正式筛选路径的响应速度与故障隔离能力。

## 非目标

- 不直接把主环境升级到 `czsc 1.0.0rc8`。
- 不计算上游全部 250 多个信号。
- 不对全 A 股票池持续计算分钟级 CZSC 结构。
- 不修改竞价 Top3 的模型、训练特征或排序。
- 不让 rc8 信号直接创建预警、模拟订单或真实订单。
- 不改变现有 `confluence_score`、`screening status` 或自选股风险动作定义。
- 不接入真实券商交易通道。
- 不承诺两个 CZSC 版本的结构坐标或信号逐项完全一致。

## 已确认决策

1. 采用双运行时隔离，不在同一个 Python 环境加载两个同名 `czsc` 包。
2. `0.10.12` 保持正式引擎；`1.0.0rc8` 作为独立研究引擎。
3. 工作台允许立即展示上游研究信号。
4. 筛选器只生成影子分数和影子排名，验证通过前不参与正式决策。
5. 首期策略聚焦趋势延续：日线环境、60m/30m 方向和中枢位置、5m 二买/三买确认。
6. 一买和底背驰保持观察用途，不进入首期买入评分。
7. rc8 成为正式结构引擎与 `czsc_score_v2` 进入正式排序是两个独立晋级开关。

## 方案比较

### 方案 A：双运行时隔离，采用

主 API 保留稳定依赖，rc8 安装到第二个虚拟环境，并由常驻本地 worker 处理研究请求。两套引擎消费同一份已收盘 K 线并输出项目模型。

优点：可以同数据并行比较、故障隔离和即时回退；不会让 rc8 内部 API 泄漏到业务层。缺点：镜像更大，需要 worker 生命周期和协议管理。

### 方案 B：单运行时兼容适配，不采用

每次部署只安装一个 CZSC 版本，由兼容适配器处理 API 差异。实现较少，但不能在同一运行中做真实影子对照，回退必须重新部署。

### 方案 C：继续 0.10.12 并移植信号，不采用

从上游复制部分信号定义到项目。短期运行风险低，但会形成长期维护分叉，难以持续验证与上游语义一致。

## 总体架构

```text
                  closed, normalized bars
                           |
                 ChanlunEngineCoordinator
                    /                 \
                   /                   \
      LegacyChanlunEngine          CzscRc8WorkerClient
       czsc 0.10.12                 JSONL over stdio
       main API venv                       |
            |                     isolated rc8 venv
            |                       CzscRc8Worker
            |                              |
      formal analysis              research analysis
            |                              |
            +---------- normalized project contracts
                           |
                 +---------+----------+
                 |                    |
          /chanlun workbench    shadow screener job
          formal + research     score, rank, report
```

### `LegacyChanlunEngine`

- 包装当前 `ChanlunAdapter`。
- 继续负责正式分型、笔、线段、中枢、背驰、买卖点和共振结果。
- 继续作为正式筛选、预警、回放和模拟盘的唯一结构来源。
- 对外契约保持兼容，避免本次工作引起无关页面回归。

### `CzscRc8Worker`

- 使用独立 Python 解释器，仅安装固定的 `czsc==1.0.0rc8` 及其运行依赖。
- 通过版本化 JSON Lines 协议从标准输入读取请求、向标准输出写响应。
- 接收服务端生成的标准 bar 和固定目录 ID，不接受任意 Python 模块、函数名或代码。
- 使用 `CZSC.update`、`CzscSignals.update_signals` 或对应公开增量接口逐根处理闭合 K 线。`1.0.0rc8` 没有公开的 Python 方法可在独立初始化多周期数组后直接 prime trader 信号，因此大小级别中枢共振由窄适配器在 rc8 `CZSC`/`ZS` 对象上复现上游注册函数的同一算法，并用同步收盘 fixture 做逐值对照。
- 输出项目定义的研究结构、信号证据和诊断数据，不序列化上游内部对象。原始 key/value 同时携带 rc8 `Signal` 提取的结构化 `v1`、`v2`、`v3` 和 `score`，主 API 业务规则不解析审计字符串。
- 日志只写标准错误，避免污染协议流。

### `CzscRc8WorkerClient`

- 在 API 进程内管理一个常驻 worker 子进程和带优先级的串行请求队列。
- 为请求添加协议版本、请求 ID、超时和输入大小限制。
- 校验 worker 返回的 schema、版本、时间顺序、数值范围和输入快照 ID。
- 幂等请求在 worker 意外退出后允许重启并重试一次。
- worker 不可用时返回研究层状态，不抛出到正式分析路径。

首期只使用一个 worker。一个协议请求只包含一只标的的四周期数据，筛选批次被拆成逐标的工作单元。工作台请求优先于后台筛选，并可在两个候选工作单元之间插队。是否增加第二个 worker 必须由性能基准证明有必要后再决定。

### `ChanlunEngineCoordinator`

- 验证并冻结输入 bar 快照。
- 将正式请求交给 Legacy 引擎，将研究请求交给 rc8 worker。
- 统一结果状态、缓存键、版本信息和来源诊断。
- 正式分析永远不等待 rc8 结果。
- 对照模式可生成结构数量、最后一笔方向、中枢区间和信号一致性诊断，但首期不以 rc8 结构替换主图结构。

## 双运行时打包

Docker 中保留现有 `/opt/strong-stock-api-venv`，新增 `/opt/czsc-rc8-venv`。两个构建环境分别执行：

```text
/opt/strong-stock-api-venv/bin/python -c "import czsc; assert czsc.__version__ == '0.10.12'"
/opt/czsc-rc8-venv/bin/python -c "import czsc; assert czsc.__version__ == '1.0.0rc8'"
```

镜像构建必须在 rc8 wheel、原生扩展或动态库缺失时失败，避免发布一个声称支持研究引擎但实际不可用的镜像。镜像运行时即使 rc8 worker 启动失败，主 API 仍使用稳定环境启动并在健康检查中报告研究引擎不可用。

本地开发未安装 rc8 环境时允许稳定路径运行，研究端点返回 `unavailable`。测试使用协议级 fake worker，不要求每个单元测试进程都加载两个真实版本。

## 输入数据与时间规则

### 统一 bar 输入

- 两个引擎消费完全相同的 `KlineBar` 快照。
- 日线、60m、30m 和 5m 仍使用现有数据源、分钟归档和交易时段聚合逻辑。
- bar 必须按时间严格递增，OHLCV 有限且满足价格关系，重复时间戳在进入引擎前拒绝。
- 不同复权口径不可混合；发生无法修复的复权断点时，相关周期研究状态为 `adjustment_mismatch`。

### 闭合规则

- 盘中日线环境使用上一交易日的最后闭合日线；当日 15:00 后才允许把当日日线加入研究评分。
- 分钟周期只消费现有聚合器判定为闭合的 bar。
- 未闭合 bar 可继续用于正式工作台的观察显示，但不能发送给 rc8 研究评分、回放或五年验证。
- 每个研究响应携带 `calculated_at`、`last_closed_bar_at`、`input_snapshot_id`、`adjustment_mode`、`engine_version` 和 `catalog_version`。
- 数据新鲜度以决策时点“理论上应当已经闭合的最后一根 bar”为准，不要求各周期最后时间落在同一自然日。

### 首次计算与增量更新

- 冷启动按时间顺序重放历史闭合 bar，建立 rc8 状态。
- 热更新只追加比当前 `last_closed_bar_at` 更新的闭合 bar。
- 收到不同输入哈希、历史修订、复权变化或目录版本变化时，丢弃内存状态并从快照重建。
- 不持久化不可审计的 Rust/Python 内部状态；worker 重启只依赖 bar 快照恢复。

## 标准化证据模型

项目新增 `CzscSignalEvidence`，核心字段如下：

- `id`：由目录 ID、周期、有效值和首次观察收盘时间生成的稳定 ID。
- `catalog_id`、`catalog_version`。
- `family`：`trend_context`、`second_buy`、`third_buy`、`zone_confluence`、`divergence` 或 `sell_risk`。
- `role`：`primary`、`confirmation`、`risk` 或 `observation`。
- `direction`：`bullish`、`bearish` 或 `neutral`。
- `period`，以及多周期信号的 `higher_period`、`lower_period`。
- `occurred_at`：首次观察到有效状态的闭合 bar 时间。
- `last_closed_bar_at`：本次计算使用的最后闭合 bar。
- `signal_name`、固定 `params`、`raw_key`、`raw_value`。
- `reason`：由项目映射表生成的简明中文解释。
- `input_snapshot_id`、`engine_version`、`rule_version`。

业务逻辑根据 `catalog_id` 和结构化有效值判断，不解析上游中文字符串。原始 key/value 仅用于审计和排障。

### 状态与事件

每次闭合 bar 计算当前信号状态。只有同一目录项从“其他/无效”切换为目标有效值时才创建事件。有效状态持续期间不重复发事件；先失效后重新触发可生成新事件。

事件时间必须是首次可观察到该结果的 bar 收盘时间，而不是后来才确认的历史分型时间。后续结构修订不改写已保存事件，只能生成新的快照或事件。

同一 `catalog_id + params + period(s) + raw_value + occurred_at` 只能保存一次。

## 研究 API 契约

### 单标的研究信号

```text
GET /api/chanlun/stocks/{symbol}/research-signals?lookback=220
```

返回 `CzscResearchSnapshot`：

- `status`：`ready`、`pending`、`stale`、`unavailable`、`insufficient_bars` 或 `adjustment_mismatch`。
- 四周期最后闭合时间和数据新鲜度。
- 当前有效状态、最近事件和标准化证据。
- `engine_version`、`catalog_version`、`input_snapshot_id`。
- rc8 结构诊断与 Legacy 对照摘要。
- worker 健康状态和非敏感错误说明。

该接口复用现有 `ChanlunAnalysisService` 已取得并归档的闭合 bar，不再次调用 TickFlow 或 TDX。前端与正式工作台请求并行发起；研究请求失败不影响正式请求。

缓存未命中时接口等待最多 3 秒。计算完成则直接返回 `ready`；超过 3 秒则让同一任务继续在后台运行并返回 `pending`，前端使用短轮询读取相同缓存键，不重复排队。

### 筛选影子任务

`CzscShadowScheduler` 注入现有筛选器。正式筛选器确定当前 20 至 60 只的缠论增强池后，将不可变的候选、正式基线排名和 bar 快照 ID 提交给 scheduler，并立即继续返回正式结果。

正式筛选响应新增可选顶层字段：

- `czsc_v2_job_id`
- `czsc_v2_status`：`pending`、`ready`、`partial` 或 `unavailable`

每个候选的 v2 字段初始为 `null/pending`。前端使用以下端点轮询后台结果：

```text
GET /api/chanlun/screening/shadow/jobs/{job_id}
```

任务结果包含完整增强池的分数、可用性、影子排名和证据摘要。前端只按 symbol 合并字段，不重排正式列表。`GET /api/screen/runs/latest` 可合并输入快照完全匹配的最新影子结果，但同样不得改变正式顺序。

## `/chanlun` 工作台设计

### 页面组织

保留股票搜索、四周期状态轨和 720px 主图。图表下方将现有长页面整理为三个任务标签页：

1. `分析证据`：正式确认信号、rc8 研究信号和多周期共振。
2. `回放验证`：历史回放和绩效回测。
3. `预警模拟`：预警记录、模拟订单与人工确认。

所有现有能力保留。标签页只改变组织方式，不改变预警或订单规则。

### 图层与标记

- 正式笔、线段、中枢、背驰和买卖点仍是默认主图结构。
- 增加“上游研究信号”开关，工作台默认开启；均线继续默认关闭。
- 二买、三买显示在 K 线下方；卖类信号和顶背驰风险显示在上方。
- 图上只使用 `2B`、`3B`、`顶` 等短标记。
- 同一根 K 线、同一侧的多个研究信号聚合为一个标记，例如 `3B +2`。
- 完整信号名、角色、参数、时间、来源和解释只在 tooltip 与证据列表显示。
- rc8 证据始终显示“研究”来源，不使用正式信号的视觉标识。
- 缩放后标签保持聚合，不因可视区变化改变事件坐标。

### 加载状态

正式 K 线首先渲染。研究请求独立显示加载、可用、过期或不可用状态，不使用遮罩覆盖主图，也不保留无限 spinner。单次超时后保留正式页面并允许刷新。

## `czsc_score_v2`

现有 `ChanlunScreeningSummary.confluence_score`、筛选条件和正式排序继续工作。新增字段：

- `czsc_score_v2: int | null`
- `czsc_v2_eligible: bool`
- `czsc_v2_shadow_rank: int | null`
- `czsc_v2_evidence: list[CzscSignalEvidenceSummary]`
- `czsc_v2_status`
- `czsc_v2_rule_version`，首版为 `czsc-score-v2-rule-1`

### 正向分值

| 维度 | 规则 | 上限 |
| --- | --- | ---: |
| 日线环境 | `bi-status` 向上 +12；`bi-base` 向上中继 +8、向上转折 +4 | 20 |
| 60m 结构 | 向上 +10；向上中继 +5；1d/60m 看多中枢共振 +5 | 20 |
| 30m 结构 | 向上 +10；向上中继 +5；60m/30m 看多中枢共振 +5 | 20 |
| 5m 触发 | 首要二买 +20 或首要三买 +25，匹配的辅助信号再 +5 | 30 |
| 方向共振 | 日线、60m、30m 同时向上 | 10 |

同一维度按上限截断，二买和三买不重复累加。

### 风险扣分

- 日线顶背驰或卖类风险：扣 30。
- 60m 顶背驰或卖类风险：扣 20。
- 30m 顶背驰或卖类风险：扣 10。
- 5m 当前卖类信号：扣 20。

同一周期存在多个风险时只应用该周期最大扣分，避免重复表达同一结构。总分限制为 0 至 100。

### 可用与准入

任一必需周期缺数、过期、使用未闭合 bar、复权不匹配或 worker 异常时，`czsc_score_v2` 为 `null`，不能以 0 分代替。

`czsc_v2_eligible` 仅在以下条件全部成立时为真：

1. 四周期均为新鲜闭合数据。
2. 日线没有当前顶部风险。
3. 60m 和 30m 没有当前卖类信号。
4. 最新闭合 5m 的首要信号为二买或三买。

影子排名在同一增强池内按 `eligible 优先 -> score 降序 -> 正式基线排名` 生成。它只用于报告和页面对照。

## 五年数据集

### 数据来源职责

- 运行时继续使用 TickFlow、现有分钟 SQLite 和可选 mootdx 补数。
- 五年研究允许使用现有 `free-stockdb` 服务作为离线历史数据提取源，但它不能成为 Docker 或实时筛选依赖。
- 数据提取通过配置的 `FREE_STOCKDB_BASE_URL` 访问，不在代码中硬编码局域网地址。
- free-stockdb 不可用时可使用已归档的本地历史文件或 mootdx；报告必须记录实际来源，不允许静默替换。

### 候选范围

每个历史交易日只使用当时可见数据重建现有候选池和静态过滤。随后只为进入同一 20 至 60 只增强池的标的读取所需 5m 历史，并由 5m 聚合 30m、60m。日线使用同源或可验证复权一致的历史日 K。

不下载和计算全市场五年分钟结构。历史提取按 symbol/month 写入分区 Parquet，重复日期复用已有分区。

### 数据清单

每个数据集包含：

- 起止日期、交易日历和候选构建规则版本。
- 每个分区的来源、抓取时间、复权口径、行数、最早/最晚时间和 SHA-256。
- 缺失、停牌、重复、异常 OHLCV 和除权断点统计。
- CZSC 引擎、目录、评分和交易成本版本。

源数据不足或无法统一复权的 symbol/date 不生成评分样本，并在质量报告中单独计数。

## 时间序列验证

### 样本与成交口径

- 15:00 使用当日全部已闭合数据形成候选和影子分数。
- 下一交易日开盘作为入场价。
- 第三个交易日收盘作为主要退出价，同时报告 1 日和 5 日敏感性结果。
- 停牌、无法成交的一字涨停或缺失下一开盘价记录为未成交，不用后续价格代替。
- 基线与 v2 使用完全相同的成交可用性、价格和成本。
- 默认净收益使用 20 bps 往返成本；报告同时输出毛收益，CLI 参数和清单记录实际成本。

### 滚动切分

覆盖最近五年，按以下窗口每 6 个月滚动一次：

```text
24 months development -> 6 months validation -> 6 months out-of-sample test
```

首版评分权重在进入验证前固定，不利用测试窗口调参。开发和验证窗口只用于检查阈值稳定性、数据质量和实现错误；所有晋级指标只汇总未参与调整的样本外窗口。

### 对照组合

- 基线：现有正式筛选顺序。
- v2：相同候选池中的影子顺序。
- 主报告分别比较每日 Top5 和 Top10；另输出 Top3 敏感性结果。
- 组合按每日一份等权 tranche 建仓，持有三日；同时存在的 tranche 平分组合资金。
- 两组使用相同风险过滤、成交成本和缺失处理。

### 指标

- 有效样本和未成交样本数。
- 3 日净收益胜率。
- 平均盈利、平均亏损及其绝对值比率。
- 平均、中位数和累计净收益。
- 相对基线的配对超额收益。
- 时间序列最大回撤。
- Top5/Top10 命中率、收益分布和行业集中度。
- 每个滚动区间及最近六个月的独立结果。

### 排序晋级门槛

`czsc_score_v2` 进入正式排序必须同时满足：

1. 样本外有效样本不少于 300。
2. 3 日净收益胜率大于 50%。
3. 平均盈利/平均亏损绝对值不低于 1.3。
4. Top5 和 Top10 的样本外净收益均高于当前筛选器。
5. Top5 和 Top10 的时间序列最大回撤均不差于基线。
6. 最近六个月净收益为正，且胜率相对此前样本外下降不超过 5 个百分点。
7. 每个滚动区间的未来函数、数据质量和成交一致性校验均通过。

验证程序只能生成“保持影子”或“建议晋级”，不能自动修改运行时开关。

## 可复现 CLI 与产物

新增离线 CLI。分区 Parquet 使用只安装在 `research` 依赖组的 `pyarrow`，不进入主 API 或生产 Docker 环境：

```text
uv run --project apps/api --group research python scripts/czsc-research.py build-dataset \
  --start 2021-07-01 --end 2026-06-30 --output artifacts/chanlun-v2/datasets

uv run --project apps/api --group research python scripts/czsc-research.py validate \
  --dataset artifacts/chanlun-v2/datasets/<dataset-id> \
  --output artifacts/chanlun-v2/reports
```

构建数据集和验证是两个独立步骤。验证过程不访问行情网络，只读取已冻结的数据集。

每次验证输出：

- `manifest.json`
- `metrics.json`
- `folds.json`
- `samples.csv.gz`
- `report.html`
- `checksums.sha256`

报告记录 Git 提交、Python/CZSC 版本、目录和评分版本、CLI 参数、数据集 ID、随机种子和生成时间。相同提交、数据集和参数必须生成相同的指标 JSON 与样本文件哈希；HTML 的生成时间不参与结果哈希。

## 缓存与持久化

研究运行时数据保存在 `data/chanlun/research.sqlite3`：

- `research_snapshots`：输入快照、引擎、目录、状态和时间。
- `signal_evidence`：去重后的标准化事件。
- `shadow_batches`：筛选批次、正式基线排名和后台任务状态。
- `shadow_scores`：每个候选的分数、可用性、影子排名和证据摘要。

缓存键至少包含：

```text
symbol + period(s) + last_closed_bar_at + adjustment_mode
+ input_hash + engine_version + catalog_version + score_rule_version
```

同一缓存键只允许一个在途任务。调用方复用已有 future/result，避免页面刷新和筛选任务重复计算。失败、空响应和部分结果不能保存为 `ready`。

保留旧版本快照用于审计，但只有与当前输入和版本完全匹配的快照可进入当前页面。任何不匹配都返回 `pending`、`stale` 或 `unavailable`，不能复用旧分数。

运行时研究快照和影子批次保留 180 天，去重后的信号证据保留 730 天；每个引擎/目录版本的最新成功快照不因普通清理被删除。五年研究数据和报告位于离线 artifacts，不写入运行时 SQLite。

## 故障处理

- 单标的交互请求等待上限为 3 秒；超时后任务可在后台继续，主图继续显示正式结构。
- 单标的 worker 执行硬上限为 10 秒；超过上限终止并重启 worker，该请求返回 `unavailable`。
- worker 退出：重启并对幂等请求重试一次；再次失败后返回 `unavailable`。
- 连续三次协议级失败：熔断 60 秒，期间不继续创建 rc8 请求。
- 非法 JSON、schema 不符、版本不符、时间倒序或非有限数值：拒绝整份响应，不保存部分证据。
- 筛选批任务部分失败：成功候选保存，失败候选为 `null/partial`，正式顺序不变。
- rc8 引擎不可用：正式工作台、现有缠论摘要、筛选、预警和模拟盘继续工作。
- 数据过期：允许展示最近研究快照并标记 `stale`，但不生成可用分数。

健康检查分别报告：

- Legacy 引擎版本和导入状态。
- rc8 worker 版本、协议版本和进程状态。
- 当前目录版本。
- 最近成功时间、连续失败数和最近非敏感错误。

## 性能预算

1. 正式工作台和正式筛选路径 P95 相对基线回归不超过 5%。
2. 单标的研究快照缓存命中 P95 小于 150ms。
3. 单标的四周期冷计算 P95 小于 2 秒。
4. 60 只候选、四周期影子批次在目标 Unraid CPU 上小于 30 秒，且全程后台执行。
5. 60 只候选基准期间 worker RSS 不超过 1 GiB；批次完成后释放非活跃 symbol 状态，并回落到批次前 RSS 的 120% 以内。
6. 不使用 GPU，不执行全市场分钟级实时计算。

如果第 4 项未通过，优先优化批协议、输入去重、缓存和 worker 生命周期。没有基准证明前不增加进程池，也不通过减少闭合校验来换取速度。

## 测试设计

### 单元测试

- 两版本 bar 转换、频率映射和标准化结构契约。
- 每个目录项的参数白名单、有效值映射、角色和中文解释。
- 状态转事件、失效后重触发和同 bar 去重。
- `czsc-score-v2-rule-1` 的正向分、维度上限、风险扣分、空分和 eligible 条件。
- 乱序、重复、非法 OHLCV、复权不匹配和未闭合 bar 拒绝。
- 缓存键、版本失效、并发请求合并和旧快照隔离。

### 协议和集成测试

- worker 正常响应、超时、退出、非法 JSON、schema 错误、版本错误和重启恢复。
- rc8 故障时正式分析 API、工作台、筛选顺序、预警和模拟订单不变。
- 正式筛选先返回，影子任务随后完成并只按 symbol 合并字段。
- 影子批次单标的失败不影响其它候选。
- Docker 两个解释器分别导入正确版本，运行时健康状态准确。

### 无未来函数测试

对真实固定 fixture 和合成边界数据执行两种计算：

1. 从第一根 bar 开始逐根增量更新。
2. 对每个时间点重新截断完整序列并计算。

同一时间点首次可见的信号状态和事件必须一致。任何需要后续 bar 才确认的结构不能回填为更早事件。五年验证在每个滚动窗口抽样重复该检查。

### 前端和视觉测试

- 正式图层与研究图层可独立开关。
- 同一 K 线多信号聚合为单一标记，tooltip 完整展示证据。
- 缩放、周期切换、窄屏和长股票名称下不发生文字重叠。
- 研究加载、过期、失败和空状态不遮挡 K 线。
- 标签页保留回放、回测、预警和模拟盘原有操作。
- 使用 Playwright 在桌面和移动视口截图检查主图尺寸、标记位置、无重叠和非空渲染。

### 性能测试

- 记录正式路径改动前后的固定 fixture P50/P95。
- 分别测量缓存命中、单标的冷启动和 20/40/60 候选批次。
- 性能结果写入 CI artifact 或本地验证报告，不以人工感觉判断通过。

## 分阶段交付

### 阶段 1：协议、双环境与研究证据

- 建立 worker 协议、双适配器和标准化证据模型。
- 固定信号目录和无未来函数 fixture。
- 增加研究端点、缓存与健康状态。
- 正式结构和正式筛选保持不变。

### 阶段 2：工作台与影子筛选

- 增加研究标记、证据列表和工作台标签页。
- 增加评分规则、后台影子批次和候选字段。
- 验证正式路径性能没有明显回归。

### 阶段 3：五年数据集与验证

- 建立离线历史适配器、分区数据集和质量清单。
- 实现滚动验证、组合对照和可复现报告。
- 根据晋级门槛生成结论，不自动切换。

### 阶段 4：独立晋级评审

- 结构引擎评审：契约、增量一致性、图表视觉、性能和回放全部通过后，才讨论 rc8 成为正式结构来源。
- 排序策略评审：五年样本外门槛全部通过后，才讨论 `czsc_score_v2` 进入正式排序。
- 任一评审失败都保持当前稳定路径，不影响另一条研究工作。

## 验收标准

1. 两个 CZSC 版本可在同一部署中隔离运行，且健康状态明确。
2. rc8 只消费已闭合、可审计、复权一致的 bar。
3. 工作台正式主图先显示，研究层失败不影响其使用。
4. 图上研究标记简洁、聚合正确，并能追溯到完整证据。
5. 正式筛选结果先返回，影子任务不改变正式顺序和筛选状态。
6. 缺数、过期和失败候选的 v2 分数为 `null`。
7. 固定 fixture 的逐根与截断计算通过无未来函数对照。
8. 正式路径、单标的和批次性能满足预算。
9. 五年 CLI 在冻结数据集上可复现，并生成完整质量、折叠、交易和指标产物。
10. 任何正式晋级都需要门槛报告和人工确认，不存在自动开关。
