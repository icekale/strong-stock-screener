# 首页首屏性能优化设计

## 目标

优化 React 首页 `/` 的首屏数据呈现速度和重复请求行为，不改变市场数据、竞价模型、情绪计算或其他业务 API 的语义。首屏应优先呈现市场状态，板块资金流和趋势数据在后台独立加载；用户进入其他页面后返回首页时，短时间内应复用已经取得的结果。

## 当前边界

- 目标实现位于 `apps/web`，不修改仍保留的 `apps/web-vue` 旧实现。
- 首页核心请求为市场概览、板块资金流和情绪摘要；板块轮动与盘中情绪曲线已经通过可视区域激活后加载。
- API 服务和响应结构保持不变，不新增聚合接口。
- 当前工作树中已有用户未提交改动，实施时只修改首页性能相关文件，不覆盖或格式化其他改动。

## 方案

### 1. 分层加载状态

`MarketOverviewWorkbench` 将核心请求与后台请求分开管理：

- 市场概览作为首屏关键数据，完成后立即显示指数、成交额、涨跌家数和涨停跌停。
- 板块资金流、情绪摘要继续并行请求，但不阻塞市场状态的可用状态，也不让慢请求持续占用首页的全局刷新状态。
- 趋势区仍由 `IntersectionObserver` 激活，并保持独立的请求代次保护，避免旧请求覆盖新数据。
- 各面板保留现有 loading、error、stale 和 empty 状态。

### 2. 短时内存缓存与请求去重

在首页数据加载边界增加专用的内存缓存：

- 缓存键包含交易日、接口参数和数据面板名称，避免不同日期或参数复用错误数据。
- 默认缓存窗口为 15 秒，适用于页面切换返回和组件重新挂载；不写入 `localStorage` 或 `sessionStorage`，避免硬刷新后展示旧行情。
- 同一缓存键存在进行中的请求时，复用同一个 Promise，避免重复请求和重复触发数据源计算。
- 显式点击“刷新”绕过已完成缓存，但仍受到最新请求代次保护；刷新结果成功后更新缓存，失败时保留已有面板数据并显示 stale/error 状态。
- 缓存只负责客户端请求协调，不改变后端 TTL、数据源优先级或数据质量标记。

### 3. 图表与非首屏资源

- 保持趋势图的按需 ECharts 动态导入，不把图表初始化加入关键首屏路径。
- 趋势数据仅在趋势区域接近视口时请求；已激活后刷新趋势不重新触发首页核心请求。
- 不引入新的首页图表或装饰性动画。

## 数据流

```text
进入首页
  -> 读取内存缓存
     -> 命中：立即提交对应面板状态
     -> 未命中：创建或复用请求 Promise
  -> 市场概览完成：首屏市场状态可用
  -> 板块资金流 / 情绪摘要各自完成：独立更新对应面板
  -> 趋势区进入视口：请求板块轮动和情绪曲线
```

所有请求继续使用已有的 `PanelState` 和 `executeLatestOnly` 机制。组件卸载时不取消共享请求，但不会向已卸载组件提交状态；下一次挂载可以复用请求或短时缓存。

## 验收指标

使用本地浏览器和网络请求日志对比优化前后：

1. 首页首次进入时，市场状态不再等待板块、情绪或趋势请求完成；任一核心请求失败不阻塞其他面板。
2. 同一页面生命周期内重复进入首页或重复触发相同请求时，缓存窗口内不产生重复 HTTP 请求。
3. 连续点击刷新不会出现旧响应覆盖新响应，刷新按钮状态不会被后台趋势请求长期占用。
4. 趋势区域未进入视口前，不发起趋势接口请求，也不初始化 ECharts。
5. 首页现有数据状态、链接、布局和 API 响应结构保持不变。
6. `apps/web` 类型检查、单元测试、生产构建和 `git diff --check` 通过；桌面与移动端首页无横向溢出。

## 测试策略

- 为缓存协调逻辑增加单元测试：缓存命中、TTL 过期、同键并发去重、显式刷新绕过缓存、失败不污染成功缓存。
- 为首页加载流程增加源码/行为契约测试：核心面板独立提交、后台趋势保持视口激活、刷新代次保护和缓存调用边界。
- 运行现有 `apps/web` 测试与类型检查，随后执行生产构建。
- 启动本地首页进行桌面 `1280x720` 和移动 `390x844` 浏览器检查，核对首屏状态、请求数量和页面无横向溢出。

## 不在本次范围

- 不修改后端接口、数据源连接、后端缓存 TTL 或模型训练/选股逻辑。
- 不改造 AppShell、导航、个股 K 线、缠论、竞价和板块页面。
- 不使用持久化浏览器缓存，不引入新的状态管理库或请求库。
